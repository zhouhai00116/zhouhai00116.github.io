关系数据理论

数据依赖：函数依赖和多值依赖

关系模式存在的问题：①数据冗余②更新异常③插入异常④删除异常

规范化

函数依赖

定义：若对于R（U）——关系模式的属性集中的任意一个关系r，r中不可能存在两个元祖在X上属性值相等而在Y上缺不相等，则可称为X函数确定于Y，Y函数依赖于X

*此处的函数依赖不分特殊情况，任意时刻、全部关系都需满足`

术语和记号：

非平凡的函数依赖：X确定于Y，但Y并非X的子集

平方的函数依赖：X确定于Y且Y为X的子集

X确定于Y，则称X为此函数依赖中的决定因素>

完全函数依赖：只有X属性集全部集合才可以确定Y，任一子集都不可

X（F）Y

部分函数依赖：X属性集中存在着子集也可确定Y

X（P）Y

传递函数依赖：X可确定Y，且为非平凡函数依赖，Y可确定于Z，也为非平凡函数依赖，则可称X确定Z

码：

候选码：若U完全依赖于K，则K为关系R的候选码

超码：若U部分依赖于K，则K为关系R的超码

*候选码是最小的超码，K的任一子集都不会为候选码

主码：若候选码唯一则直接为主码，若大于一个则可以选定其中一个

主属性：任何一个候选码中的属性都为主属性

非主属性（非码属性）：非候选码的属性

若整个属性都为候选码则可称为全码

外码：R中属性非X中的码，但X为另一个关系模式的码，则X称为外码

规范化层级

1NF,2NF,3NF,BCNF,4NF.5NF

1NF

每一个分量必须是不可分的数据项

2NF

在1NF的基础上，且每一个非主属性都完全函数依赖于任何一个候选码

不属于2NF会导致的问题：

①插入异常

②删除异常

③修改复杂

3NF

每一个非主属性既不传递依赖于码，也不部分依赖于码

BCNF

在关系模式中，若每一个决定因素都包含码，则可为BCNF

结论：

①所有非主属性对每一个码都是完全函数依赖

②所有主属性对每一个不包含它的码也是完全函数依赖

③没有任何属性完全函数依赖于非码的任何一组属性

规范化小结：

1NF（消除非主属性对码的部分函数依赖）

2NF（消除非主属性对码的传递函数依赖）

3NF（消除主属性对码的部分和传递函数依赖）

BCNF（消除非平凡且非函数依赖的多值依赖）

4NF

数据依赖的公理系统

逻辑蕴涵：一组函数依赖F在关系模式R中，任意一个关系r都可以确定X确定于Y，则可称F逻辑蕴涵于X确定于Y

Armstrong公理系统推理规则：

R为关系模式，U为属性集总体，F为一组函数依赖

A1——自反律：X确定于Y为平凡函数依赖，则X确定于Y为F所蕴涵

A2——增广律：若X确定Y为F所蕴涵，则XZ确定YZ也为F所蕴涵

A3——传递律：若X确定Y，而Y确定Z均为F所蕴涵，则X确定Z也为F所蕴涵

由A1,A2,A3可推出：

①合并规则：X确定Y，X确定Z，则X确定YZ

②伪传递规则：X确定Y，WY确定Z，XW确定Z

③分解规则：X确定Y，且Z为Y的子集，则X确定Z

闭包：F所逻辑蕴涵的函数依赖的全体为F的闭包（记为F+）

Armstrong公理的有效性指：由F出发根据Armstrong公理推导的每一个函数依赖一定在F+中

完备性：F+中的每一个函数依赖，必定可以由F出发根据Armstrong公理推导出来

X确定A能由F根据Armstrong公理导出，则称X（F）+为属性集X关于函数依赖集F的闭包

算法6.1：

找出初始给定的属性集X，在F中找出左边为X的子集所确定的集合，再合并起来成一个新的属性集

若F+=G+，则F与G等价

最小依赖集（最小覆盖）F(m)：

①F中任一函数依赖的右部仅含有一个属性

②F中不存在F与F-X确定A等价（即不存在着与其它函数依赖闭包重复的函数依赖）

③F中不存在X确定A，X有真子集Z使得F-X确定A并Z确定A等价F（即不存在着可由其子集来替代的函数依赖闭包）

找出最小依赖集：

①将F中所有函数依赖的右边化为单一属性

②去掉F中所有函数依赖左边的冗余属性

③去掉F中所有冗余的函数依赖

模式分解

三个定义：

①无损连接性

②保持函数依赖

③即有无损连接性，又要具有无损连接性

分解至3NF的模式分解算法：

合成法（可保持函数依赖）：

①对关系模式R中的函数依赖集F进行“极小化处理”

②找出所有不在F中出现的属性（记做U0），将U0牵引出的属性也一并从U中去掉

③若其中能找出X确定于A为F所蕴涵，且XA=U，则算法终止

④否则，对F按具有相同的左部的原则分组，每一组函数依赖所涉及的属性构成Ui，若Ui为其它Uj的子集则去除。此时，所有的Ri均为3NF

转换为3NF即有无损连接性又保持函数依赖性的模式分解：

①在合成法的基础上，已将R分解成保持函数依赖的3NF，设X为原先R的码，则构建一个新的集合C为P并上X在R中的关系模式

②若存在Ui为X子集或反之，则将X在R中的关系模式去除

③C就位既有无损连接性又保持函数依赖性的模式分解

模式分解推论：

①分解保持函数依赖就满足3NF

②分解即有无损连接线又保持函数依赖，可3NF，但不一定BCNF

③分解具有无损连接线，则可4NF

数据库设计

概述

​ 数据库设计：构造优化的数据库逻辑模式和物理结构，并据此建立数据库及应用系统，使之能够有效地存储和管理数据，满足各种用户的应用需求，包括信息管理要求和数据操作要求。

步骤：需求分析；概念结构分析；逻辑结构分析；物理结构分析；数据库实施；数据库运行和维护

数据字典：数据项；数据结构；数据流；数据存储；处理过程；

概论设计（E-R图）

​ 矩形：实体；椭圆：属性；菱形：关系（大部分均为强实体）

​ *：双矩形代表着实体为弱实体，对应的关系也为弱关系–双菱形表示

​ *：弱实体拥有完全的依靠性

​ 实体间的联系分为三类：1:1;1:n;m:n;

​ 联系的度：n个实体间的联系度就为n(n>=2)

​ 拓展的E-R图：

​ ISA联系:

​ (1)分类属性：存在父类与子类

​ （2）不相交约束和可重叠约束:父类的一个实体不可为子类的重叠

​ （3）完备性约束：倘若父类的实体完全是某一子类实体–完全特化（双线）；若有超出–部分特化（单线）

​ 基数约束：数字针对的关系联接的另一实体

​ Part-of联系：强弱实体型

​ 设计准则：①属性不可再被描述②属性不能与其它实体具有联系

​ E-R图间的冲突：①属性冲突：值的冲突②命名冲突③结构冲突：不同E-R图对于同一对象的定义结构不同

​ E-R图设计的步骤：①从分E-R图中进行合并②消除冲突和冗余（可采用规划化理论）

*联系也可以拥有属性

逻辑设计

E-R图向关系模型转换：①一个实体型可直接转化为一个关系模式②1:1；1：n；n:m均可转化成关系模式③拥有相同码的关系模式可合并

模型优化：①数据依赖②消除冗余③分析范式④环境分析⑤分解：水平分解–对元祖本身分解为小元祖；垂直分解：将属性分解为若干子集合

物理设计

对数据库进行物理结构确定及评价：运行评率和性能要求

存取方式：索引方式（B+和Hash）和聚簇（集中存放）

*一个基本表只能建立一个聚簇

数据库完整性

定义：指数据的正确性、相容性（为了防止数据库中存在不符合语义的数据）

为进行数据库完整性所需的机制：

1.提供定义完整性约束条件的机制（实体完整性、参照完整性、用户定义完整性）

2.提供完整性检查的方法

3.进行违约处理（NO ACTION拒绝操作、CASCADE级联处理）

实体完整性

定义：

有两种约束方法：1.列级约束条件–在定义行后直接定义 2.表级约束条件——在建表最后

二者之间的区别：

列级约束条件只可适用于码为单属性的关系模型

表级约束条件无论码为多属性或单属性都可以

检查和违约处理：

①检查主码值是否唯一

②检查主码的各个属性是否为空

一般为了提高效率，通常采用B+树（理论上来说是B-树，关键字和分支为n:n+1）索引

参照完整性

定义：

找出关系模式中为外码的属性并定义

格式为：foreign key(xx) references xx(xx)

检查和违约处理：

有四种可能会破坏参照完整性的情况：

①主表中增加一个元祖，但在另一个表中，外码的属性找不到一个相同的元祖值

②主表中修改一个元祖，但在另一个表中，外码的属性找不到一个相同的元祖值

③外码表中删除一个元祖，但在主表中，外码的属性找不到一个相同的元祖值

④外码表中修改一个元祖，但在主表中，外码的属性找不到一个相同的元祖值

在主表中删除是不影响的，在外码表中增加同样也是不影响的

处理策略：

①NO ACTION拒绝该操作

②CASCADE级联操作——将所有影响到参照完整性的数据都变动

③设置为空值

拒绝：on xxx(操作类型,insert——只针对主表，update——两个表都可以，delete——只针对副表) no action

级联：on xxx(操作类型,insert——只针对主表，update——两个表都可以，delete——只针对副表) cascade

设空: on xxx(操作类型,insert——只针对主表，update——两个表都可以，delete——只针对副表) set null

用户定义的完整性：

定义：

①不允许空值 not null

②列值唯一 union

③用check指定列值应该满足的条件

check（）后面可以跟语句表达式

属性或元祖检查和违约处理:

在插入或修改元祖的值时就会进行检查，若违约则直接拒绝操作

属性级的约束为列级约束

元祖级的为表级约束

完整性约束命名子句（便于对约束进行一统操作）：

定义：constraint<完整性约束条件名><完整性约束条件>

在条件中可以书写not null、unique、primary key、foreign key、check

对于完整性约束不存在修改

触发器：

定义：

create trigger<触发器名>

{before|after}<触发事件>on<表名>

referencing new|old row as<变量>

for each{row|statement}

[when<触发条件>]<触发动作体>

规则：

①只有表的拥有者才可以在表上创建触发器且有数量限制

②在同一模式下，触发器的名字是唯一的

③触发器只能定义在基本表中，不可用于临时表——视图之类的

④触发事件：可为类型的单个体也可为组合体

⑤触发器类型：行级触发器（每执行一次触发事件就触发）、语句级触发器（待全部事件执行完再触发）

⑥触发条件：当when存在时，条件为真才执行。若不存在when触发器被激活后立刻执行

⑦触发动作体：可以是一个匿名pl\sql过程块，也可是对已存在的过程块调用

触发器激活顺序：

①执行该表上的before触发器

②激活触发器的sql语句

③执行该表上的after触发器

若为同一类型遵循“FIFO”

删除触发器：

drop trigger<触发器名>on<表名>

②把投影运算和选择运算同时进行

③把投影同其前或后的双目运算结合

④选择同前面要执行的笛卡尔积结合起来为一个连接运算

⑤找出公共子表达式

物理优化

选择操作的启发式优化：

①“主码=值”——主码索引

②“非主属性=值”，元祖数目<10%——索引扫描，否则全盘扫描

③and操作的优先采用组合索引扫描方法

④or操作的优先采用全表顺序扫描

连接操作的启发式规则：

①若两表——排序—合并算法

②有索引，索引连接算法

③均不适用以上两个，且有个表较小——hash join

④最后采用嵌套循环（小外大内）

代价估算：

1.统计信息：

①基本表——元祖总数N、元祖长度L、占用的块数B、占用的溢出块数BO

②列——列不同值个数m、最大值、最小值、索引相关信息

③总索引信息

2.代价估算示例：

①全表：单位B块——cost=B;“码=值”——cost=B/2

②索引扫描：B+——cost=L+1；非码——cost=L+S(重复性元祖);若为比较元祖——平均cost=L+Y/2+B/2

③嵌套循环代价：cost=Br+BrBs/(K-1)

④排序-合并连接（有序）cost=Br+Bs+(Frs连接选择率)*Nr*Ns/Mrs
